# QuickEval V1 产品与技术规格

> 状态：需求与技术选型基线已确认  
> 更新日期：2026-07-23

## 1. 项目定位

QuickEval 是一个面向小团队的轻量级 Agent 人工评测系统，主要用于云市场平台中的智慧助手和智能采购 Agent。

V1 采用单组织模式，不考虑多租户、计费和复杂组织架构。系统以黑盒人工评测为主，不直接调用 Agent API。评测人员在真实业务系统中完成交互后，将 Agent 回答文本和产品截图录入 QuickEval，进行评分、问题定位和 Badcase 管理。

## 2. V1 目标

- 管理不同的评测对象、业务场景、评测集和评测集版本。
- 支持团队成员基于已发布评测集独立开展人工评测。
- 保存 Agent 回答文本、真实交互截图、人工评分和评语。
- 统一管理评测发现和真实业务发现的 Badcase。
- 汇总评测结果，并分析 Badcase 状态及问题标签分布。
- 保持部署、权限和操作流程轻量，为未来 OA 登录和自动评测预留扩展边界。

## 3. V1 不包含的范围

- 多租户、计费和复杂组织权限。
- 评测负责人分配任务、审批或领取流程。
- Agent API 配置、密钥管理和自动调用。
- 批量自动执行、自动评分和 LLM-as-a-Judge。
- Agent 工具调用链路、决策轨迹和白盒评测。
- 语音、图像等多模态 Agent 评测。
- OA 账号系统的实际接入。
- OCR、Elasticsearch、消息提醒和复杂工单流程。
- Docker 或 Kubernetes 部署。

## 4. 用户与权限

### 4.1 用户角色

V1 只包含两种角色：

- 普通成员：开展评测、登记 Badcase、维护自己提交的内容。
- 管理员：管理用户、评测对象、场景、评测集和全局问题标签，并可维护全部数据。

### 4.2 登录方式

- 不开放自主注册，由管理员创建账号。
- 用户使用用户名或邮箱加密码登录。
- 管理员可以重置用户密码。
- 登录会话存储在 Redis。
- MySQL 保存用户、角色和密码哈希，不保存明文密码。
- 认证模块需要提供可替换的身份认证边界，后续可接入公司 OA 账号系统。
- 业务数据只关联 QuickEval 内部用户 ID，不直接依赖邮箱、工号或 OA 标识。

## 5. 核心领域关系

```text
评测对象
└── 场景
    ├── 评测集
    │   └── 评测集版本
    │       └── 评测用例
    │           └── 人工评测结果
    └── Badcase
        ├── 问题标签
        ├── 截图
        └── 处理时间线
```

关系约束：

- 一个评测对象包含多个场景。
- 一个场景只属于一个评测对象。
- 一个场景包含多个评测集。
- 一个评测集包含多个不可变的已发布版本。
- 每次人工评测绑定一个确定的评测集版本。
- 评测产生的 Badcase 和业务登记的 Badcase 统一进入 Badcase 中心。

## 6. 功能模块

### 6.1 评测对象管理

管理员维护评测对象，例如“云市场智慧助手”和“智能采购 Agent”。评测对象至少包含名称、描述、启用状态和审计字段。

### 6.2 场景管理

- 创建、编辑、停用和查看场景。
- 创建场景时必须选择一个评测对象。
- 场景下管理评测集和 Badcase。
- 场景停用后不允许创建新的评测或 Badcase，但保留历史数据。

### 6.3 评测集与版本

评测集版本包含以下状态：

- 草稿：允许增删改用例。
- 已发布：内容锁定，可以开始评测。
- 已归档：保留历史数据，但不能开始新的评测。

版本规则：

- 已发布版本不可直接修改。
- 修改已发布内容时，复制生成新的草稿版本。
- 评测任务始终绑定具体版本快照。
- 后续版本变化不影响历史评测中的题目和上下文。

### 6.4 评测用例

每条评测用例支持以下字段：

- 用例名称。
- 用户问题或任务指令，必填。
- 前置条件或操作步骤，可选。
- 期望结果或参考答案，可选。
- 评判要点，可选。
- 用例标签，可选。
- 排序和启用状态。

开放式智慧助手不存在唯一标准答案，因此期望结果不能设为必填。

### 6.5 CSV 导入导出

- 提供 UTF-8 编码的标准 CSV 模板。
- 导出 CSV 时携带 BOM，避免中文在常用表格软件中乱码。
- 正确处理逗号、双引号和多行文本。
- 导入后先预览和校验，再由用户确认写入。
- 校验错误需要定位到具体行和字段。
- CSV 只能追加导入到草稿版本。
- V1 不提供覆盖导入；整体替换通过新建草稿版本完成。
- 支持将评测集版本导出为 CSV。

### 6.6 人工评测

- 任何已登录成员都可以选择已发布的评测集版本并开始评测。
- 不需要负责人分配、审批或领取任务。
- 每位评测人员的评测记录相互独立。
- 同一人员可以重复评测同一个评测集版本，每次形成独立记录。
- 开始评测时记录 Agent 版本、运行环境和可选备注。
- 环境使用固定选项：测试、预发布、生产、其他。
- 评测支持暂存、退出后继续和最终完成。
- 只有所有启用用例都处理完成后，整次评测才能标记为已完成。
- 未完成评测不进入正式平均分和完成率统计。

#### 6.6.1 单条用例结果

每条用例可以记录：

- Agent 回答文本，用于搜索、分析和定位。
- 多张真实产品交互截图，用于保留现场证据。
- 可选的 1～5 分点击评分，不允许自由输入分数。
- 评语。
- Badcase 标记。

评分含义：

| 分数 | 含义 |
| --- | --- |
| 1 | 完全错误 |
| 2 | 大部分错误 |
| 3 | 基本可用但有明显问题 |
| 4 | 基本正确 |
| 5 | 完全符合预期 |

Badcase 标记与评分相互独立，不根据分数自动判断 Badcase。标记 Badcase 或判定明显失败时，需要填写评语。

#### 6.6.2 跳过用例

无法执行的用例可以标记为已跳过，但必须选择或填写原因，例如环境不可用、无权限或用例不适用。

跳过项：

- 允许整次评测继续完成。
- 不计入平均分和 Badcase 比例。
- 在汇总中单独展示数量和原因。

### 6.7 Badcase 中心

Badcase 中心统一管理两种来源：

- 评测发现：关联场景、评测集版本、评测用例、评测记录、评测人员和评分。
- 业务发现：不依赖评测集，由成员选择场景后单独登记。

业务 Badcase 支持记录：

- 所属场景，必填。
- 标题和问题描述。
- Agent 回答文本。
- 多张真实业务截图。
- 运行环境。
- Agent 版本，可选。
- 问题发生时间。
- 业务单号或会话 ID，可选。
- 问题标签，可多选。
- 创建人和审计时间。

### 6.8 Badcase 标签

- 使用全局统一的问题标签库。
- 由管理员创建、编辑、排序和停用标签。
- 一个 Badcase 可以选择多个标签。
- 标签允许停用，不物理删除，避免历史统计失真。

预置标签包括：

- 需求理解错误。
- 事实错误。
- 信息不完整。
- 推荐不准确。
- 工具或流程失败。
- 响应异常。
- 交互体验问题。
- 其他。

### 6.9 Badcase 处理闭环

Badcase 状态包括：

- 待处理。
- 处理中。
- 已解决。
- 暂不处理。

每个 Badcase 支持可选负责人和处理备注。状态变更、负责人变更和新增备注组成不可覆盖的处理时间线，每条记录包含操作者和操作时间。

V1 不提供评论回复、@提醒和消息通知。

### 6.10 汇总分析

看板至少包含：

- 已完成评测次数。
- 已评用例数。
- 平均分，仅统计已评分项目。
- 1～5 分分布。
- Badcase 数量和占比。
- Badcase 问题标签分布。
- Badcase 状态分布。
- 跳过用例数量和原因分布。
- 同一评测集不同版本的结果对比。

支持按以下条件筛选：

- 评测对象。
- 场景。
- 评测集及版本。
- 评测人员。
- Agent 版本。
- 运行环境。
- Badcase 来源、状态和标签。
- 时间范围。

### 6.11 搜索与导出

使用 MySQL 提供基础关键词搜索与组合筛选，覆盖：

- 评测对象、场景、评测集和用例名称。
- 用户问题和 Agent 回答文本。
- Badcase 标题、描述和处理记录。
- 业务单号和会话 ID。

支持按当前筛选条件导出：

- 评测结果明细和汇总 CSV。
- Badcase 列表 CSV。
- Badcase 标签及状态分布原始数据 CSV。

截图不打包导出，CSV 中只提供系统内的鉴权访问链接。

### 6.12 数据保留与审计

- 草稿评测集和未提交的评测记录可以删除。
- 已发布评测集只能归档。
- 已完成评测只能作废，不能物理删除。
- Badcase 可以关闭或标记无效，不能直接物理删除。
- 用户和标签只能停用。
- 关键管理员操作保留审计记录。

## 7. 截图存储

### 7.1 存储方式

- 截图存储在程序同级的 `uploads` 目录。
- MySQL 只保存文件元数据和相对路径，不保存图片二进制。
- 文件目录使用系统 ID，实际文件名使用 UUID，避免重名和路径穿越。

目录结构：

```text
uploads/
├── evaluations/{evaluation_id}/{evaluation_item_id}/{uuid}.{ext}
└── badcases/{badcase_id}/{uuid}.{ext}
```

### 7.2 上传限制

- 允许 PNG、JPG/JPEG 和 WebP。
- 每条评测结果或 Badcase 最多上传 10 张。
- 单张最大 10 MB。
- 禁止 SVG 等可能包含脚本的格式。
- 文件必须在登录后通过 Go 后端鉴权访问，不能由 Nginx 直接公开整个上传目录。

### 7.3 存储扩展

后端定义统一文件存储接口，V1 实现本地文件存储。未来可以增加 MinIO 或 S3 兼容对象存储实现，业务模块不直接依赖本地路径。

## 8. 技术架构

### 8.1 架构形式

V1 采用前后端分离的模块化单体，不拆分微服务。

```text
浏览器
   │
   ▼
Nginx
   ├── /              Vue SPA
   ├── /api/v1/*      Go REST API
   └── /files/*       Go 鉴权文件访问
                         │
             ┌───────────┼───────────┐
             ▼           ▼           ▼
          MySQL        Redis      uploads目录
        业务数据       登录会话       截图
```

### 8.2 前端选型

- Vue 3。
- TypeScript。
- Vite。
- Element Plus。
- Vue Router。
- Pinia，仅保存用户、权限和少量全局 UI 状态。
- Axios。
- ECharts。
- Vitest。
- Playwright。

### 8.3 后端选型

- Go 1.26.x。
- Gin。
- GORM，封装在 Repository 层中。
- 显式 SQL Migration，生产环境不依赖运行时 `AutoMigrate`。
- go-redis。
- RESTful JSON API。
- OpenAPI 3 描述接口，并生成前端 TypeScript 类型。
- 结构化日志和请求 ID。

推荐的后端调用方向：

```text
HTTP Handler → Application Service → Repository → MySQL/Redis/File
```

只有数据库、文件存储和身份认证提供方等确实可能替换的边界才定义接口，避免无意义的抽象层。

### 8.4 后端模块

```text
internal/
├── auth/              登录、Session、OA认证扩展
├── user/              用户与角色
├── target/            评测对象
├── scenario/          评测场景
├── dataset/           评测集、版本、用例、CSV
├── evaluation/        人工评测与逐题结果
├── badcase/           Badcase、标签、处理时间线
├── analytics/         汇总查询与导出
├── file/              截图存储与访问
└── audit/             审计日志
```

### 8.5 MySQL

- 使用公司已有的 MySQL 服务。
- 推荐 MySQL 8.4 LTS。
- MySQL 是业务数据的唯一事实来源。
- 数据结构变更通过独立迁移命令显式执行，不在每次启动时自动修改。

### 8.6 Redis

- 使用公司已有的 Redis 服务。
- V1 用于登录 Session、登录失败次数和简单限流。
- 可选保存短期 CSV 导入预览状态。
- V1 不缓存业务列表和看板数据。
- Redis 不保存无法从 MySQL 或用户操作中恢复的业务数据。

### 8.7 未来 Python 扩展

V1 不引入 Python 后端。未来需要自动评测时，可以增加独立 Python Worker，用于：

- 调用不同 Agent 或模型 SDK。
- 批量自动执行。
- LLM-as-a-Judge。
- 文本相似度和离线数据分析。

Go 始终负责用户、权限、配置和业务数据。Python 只负责执行自动评测任务，避免两套后端同时维护核心业务规则。

## 9. Linux 虚机部署

V1 直接部署到 Linux 虚机，不使用 Docker。MySQL 和 Redis 连接公司已有服务。

### 9.1 部署目录

```text
/opt/quickeval/
├── quickeval              # Go 后端二进制
├── web/                   # Vue 构建产物
├── migrations/            # MySQL迁移文件
├── uploads/               # 截图文件
├── config/
│   ├── quickeval.yaml     # 普通配置
│   └── secrets.yaml       # 密码和密钥
└── logs/
    ├── quickeval.log
    └── quickeval-error.log
```

所有运行路径基于程序所在目录解析，不依赖启动命令的当前工作目录。

### 9.2 进程与网络

- 使用 `systemd` 管理 Go 服务的启动、停止和自动重启。
- 使用专门的 `quickeval` Linux 用户运行程序。
- Go 服务只监听本机地址。
- 对外只开放 Nginx 的 HTTP/HTTPS 端口。
- Nginx 提供 Vue 静态文件，并反向代理 API 和鉴权文件请求。

### 9.3 配置与密钥

- MySQL、Redis、Session 和文件配置保存在程序同级 `config` 目录。
- 密码和 Session 密钥保存在独立的 `secrets.yaml`。
- 密钥配置文件权限设为仅运行用户可读，例如 `0600`。
- 所有真实配置和密钥文件必须加入 Git 忽略列表。

### 9.4 日志与持久化目录

- 应用日志写入程序同级 `logs` 目录。
- 日志按日期或大小滚动，并限制保留数量或天数。
- `uploads`、`config` 和 `logs` 不随程序升级覆盖或删除。
- 程序启动时检查 MySQL、Redis和上传目录状态。

## 10. 安全基线

- 登录 Cookie 使用随机 Session ID，并启用 `HttpOnly`、`Secure` 和合适的 `SameSite` 策略。
- 所有写操作具备 CSRF 防护。
- 密码使用 Argon2id 或 bcrypt 哈希。
- 上传文件同时校验扩展名、Content-Type 和实际文件内容。
- 所有文件路径由系统生成，不信任用户传入的目录或文件名。
- 文件下载必须经过身份认证和权限检查。
- API 统一进行参数校验、错误响应和访问日志记录。
- 数据库账号遵循最小权限原则。
- 配置文件、上传目录和日志目录不能被 Web 服务器直接遍历。
